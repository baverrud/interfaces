# FPGA Interface Library — Tool Run Instructions

## EDA Tool Initialization

Before running simulation or synthesis, initialize the toolchain using batch files in `c:\cmdtools\`. Each script updates `PATH` and preserves the original in `ORIG_PATH`.

| Tool | Command | Toolchain Location |
|---|---|---|
| **Questa 2025.3** (VHDL-2019) | `c:\cmdtools\q26.bat` | `C:\altera\26.1std\questa_fse\win64` |
| **Questa 2025.2** | `c:\cmdtools\q25.bat` | `C:\altera\25.1std\questa_fse\win64` |
| **ModelSim 2020.1** | `c:\cmdtools\m20.bat` | `C:\intelFPGA\20.1\modelsim_ase\win32aloem` |
| **Vivado 2026.1** (VHDL-2019) | `c:\cmdtools\v26.bat` | `C:\Xilinx\2026.1\Vivado` |
| **Vivado 2025.2** | `c:\cmdtools\v25.bat` | `C:\Xilinx\2025.2\Vivado` |
| **Vivado 2023.2** | `c:\cmdtools\v23.bat` | `C:\Xilinx\Vivado\2023.2` |

**Tool selection rules:**
- For VHDL-2019 features: **only** `q26.bat` (Questa 2025.3) or `v26.bat` (Vivado 2026.1) support VHDL-2019.
- For any other task: all tools are fine (including `q26.bat` / `v26.bat`).

After sourcing, `vsim` / `vivado` are available for the rest of the shell session:

```batch
:: Simulation
c:\cmdtools\m20.bat
cd axi4\sv
python ../../common/scripts/engine.py sim . top_tb

:: Synthesis
c:\cmdtools\v23.bat
cd axi4\sv
python ../../common/scripts/engine.py synth . top
```

**Note:** `v23.bat`, `v25.bat`, and `v26.bat` all run `vivado -version` as their last line. This causes a redundant Vivado invocation before the actual synthesis script runs — harmless, but adds a few seconds of startup delay.

**Terminal note:** When running via tools that may escape backslashes, use forward slashes: `c:/cmdtools/q26.bat` instead of `c:\cmdtools\q26.bat`.

## Running sim.bat / synth.bat from PowerShell

The convenience wrappers (`sim.bat`, `synth.bat`) require a `.\` prefix in PowerShell because PS does not include the current directory in the command-search path:

```powershell
# Correct (PowerShell):
cd examples/axi4/sv
.\sim.bat all          # runs all testbenches with Questa (default)
.\sim.bat all xsim     # runs all testbenches with Xilinx xsim
.\synth.bat all        # synthesizes all tops with Vivado (default)

# Wrong (will fail — PowerShell):
sim.bat all            # command not found
```

**Alternative (recommended for scripts):** Call `python engine.py` directly — this avoids the `.\` prefix and works identically in PowerShell, CMD, and Linux:

```powershell
cd examples/axi4/sv
python ../../common/scripts/engine.py sim . all           # Questa sim
python ../../common/scripts/engine.py sim . all -b xsim   # Xilinx xsim
python ../../common/scripts/engine.py synth . all         # Vivado synth
```

**EDA tool initialization for engine.py runs:**
- For Questa/ModelSim, source the env first: `c:\cmdtools\q26.bat` then run engine.py
- For Vivado, source the env first: `c:\cmdtools\v26.bat` then run engine.py
- Without sourcing, `vsim` / `vivado` will not be found

Note: `c:\cmdtools\q26.bat` runs `vsim -version` as its last line. When run inside `engine.py`'s subprocess, this is harmless — the version output is discarded. However, in an interactive terminal, vsim may stay open waiting for the next command.

### Expected synthesis warnings (benign)

| Warning | Cause |
|---|---|
| `Synth 8-7137` — "both Set and reset" | Standard `always_ff @(posedge clk or negedge rst)` pattern |
| `Synth 8-7129` — "port unconnected/no load" | AXI sideband signals tied to constant 0 (awsize, awburst, awcache, awprot, etc.) — optimized away |
| `Synth 8-6014` — "unused sequential element removed" | `wcnt` register eliminated during logic optimization |
| `Synth 8-4767` — "RAM dissolved into registers" | Register file has async reset, preventing block RAM inference |
| `Synth 8-6859` — "multi-driven net" (CRITICAL) | `bvalid_r` driven by two processes in `axi4_slave.vhd` — harmless but should be refactored |

## VHDL-2019: Vivado GUI Crash — Null Ranges

**NEVER use null-range signal constraints (`(-1 downto 0)`).** Vivado's interactive waveform viewer crashes with an `EXCEPTION_ACCESS_VIOLATION` when trying to enumerate zero-width signals.

Instead, use **1-bit safe-width stubs** (`(0 downto 0)`) — these are constant-driven and synthesis propagates them away (proven zero-waste).

`alias slave is master'converse` is fine (tested independently — no crash).

### Good (works):
```vhdl
signal s : axis_t(
    tdata(31 downto 0),
    tuser(0 downto 0)      -- 1-bit stub (never null)
);

view master of axis_t is ... end view;
alias slave is master'converse;             -- fine (not a crash source)
```

### Bad (crashes Vivado GUI waveform viewer):
```vhdl
signal s : axis_t(
    tdata(31 downto 0),
    tuser(-1 downto 0)     -- null range → crash
);
```

## VHDL-2019 Tool Support — Record Constraints & Others

### Record Constraints

VHDL-2019 record constraints (`signal s : rec_type(elem(range), ...)`) have
**confirmed inconsistent support** across EDA tools:

| Feature | Questa 2025.3 (`q26`) | Questa 2025.2 (`q25`) | Vivado 2026.1 (`v26`) |
|---------|----------------------|----------------------|----------------------|
| `view` / `alias ... converse` | ✅ Works | ✅ Works | ✅ Works |
| Record constraints (≤6 elements) | ✅ Works (`axis_t` — sim tested) | ✅ Works (`axis_t` — sim tested) | ✅ Works (by spec) |
| Record constraints (10+ elements) | ❌ Fails (`axilite_t`, `axi4_t`) | ❌ Fails (`axilite_t`, `axi4_t`) | ❓ Untested |

**Root cause:** Questa's VHDL-2019 parser (both 2025.2 and 2025.3) fails with
`expecting IDENTIFIER` when the record constraint list exceeds ~6 unconstrained
`std_logic_vector` elements.

### `others` in Unconstrained Aggregates

`(others => value)` on view port elements with **unconstrained types** fails
in **both** Questa 2025.2 and 2025.3:

```
(vcom-1076) OTHERS choice cannot be used in unconstrained array aggregate.
```

**Fix:** Use a range-choice aggregate instead:
```vhdl
-- Bad (fails):
aw.awaddr <= (others => '0');

-- Good (works):
aw.awaddr <= (aw.awaddr'range => '0');
```
This was confirmed in `axi4_master.vhd` which had two such assignments at
lines 49 and 70. Replacing `(others => '0')` with `(aw.awaddr'range => '0')`
and `(ar.araddr'range => '0')` resolved all compilation errors.

### Questa Nodelocked License

Questa uses an **uncounted nodelocked license** — only one session at a time.
If a prior session left a license checked out, subsequent runs fail with:
```
Error: License checkout has been disallowed because only one session
       is allowed to run on an uncounted nodelocked license
```

**Kill stuck sessions:**
```powershell
taskkill /f /im vish.exe
taskkill /f /im vsimk.exe   # if still running
```
The most common offender is `vish.exe` (the Questa TCL shell).

## ConPTY note

Vivado `-mode batch` prints its banner then exits without running Tcl when
stdout is a ConPTY or pipe (VS Code integrated terminal). To run Vivado batch
from this terminal, use `start /wait` or write a runner `.bat`:

```bat
@echo off
call c:\cmdtools\v26.bat
cd /d <project>\vhdl
python ..\..\..\common\scripts\engine.py synth . top > C:\temp\out.txt 2>&1
```

Then execute it via: `start /wait "" cmd /c C:\temp\runner.bat`

Alternatively, use the Python snippet runner (no ConPTY issues) or source the
settings batch file directly instead of using the `.bat` wrapper:
```
set PATH=C:\Xilinx\2026.1\Vivado\bin;C:\Xilinx\2026.1\Vivado\lib\win64.o;%PATH%
set XILINX_VIVADO=C:\Xilinx\2026.1\Vivado
```

## Vivado Cruft Cleanup

Vivado and ModelSim leave journal files (`.jou`, `.log`, `transcript`, `.Xil/`) in the working directory. After any tool run:

```powershell
# Remove from current directory
del vivado*.jou, vivado*.log, transcript, vsim.wlf -EA SilentlyContinue
Remove-Item .Xil -Recurse -Force -EA SilentlyContinue
```

## Coding Style

All VHDL and SystemVerilog source files in this repository follow these conventions:

- **Indentation:** 2 spaces per level (no tabs).
- **Record types:** Signal names left-aligned within each record, colon-aligned with the type.
- **Mode views:** `-- channel` comment markers separate AW, W, B, AR, R groups. Outputs and inputs are on separate lines.
- **Section headers:** Use `-- ==== ... ====` banners for major sections.
- **Line length:** Keep lines readable — wrap long signal lists with one signal group per line.
